# 0002 – Graphiques légendés (radio/IRM) pour appuyer l’analyse

## Contexte

Dans un souci de vulgarisation, ajouter des **graphiques** (images de radio ou d’IRM par exemple) pour **appuyer l’analyse fournie par le LLM**, avec des **légendes** superposées. Les légendes doivent être générées par un **LLM qui propose les coordonnées de départ et d’arrivée des flèches** (pour dessiner des annotations sur l’image).

## Fichiers et fonctions concernés

### Backend

- **`server/prompts.js`**  
  - Ajouter un prompt système et un prompt utilisateur pour l’étape « légendes » : entrée = image (base64) + extraction (et éventuellement vulgarisation), sortie = JSON avec une liste d’annotations (label + flèche : x1, y1, x2, y2 en coordonnées normalisées 0–1).

- **`server/llm.js`**  
  - S’assurer que l’appel pour les légendes peut envoyer du contenu multimodal (texte + image) au LLM. Le code actuel gère déjà les images en base64 (Gemini et format OpenAI-like) ; réutiliser `chatCompletion` avec un message user contenant `content: [{ type: "text", text: "..." }, { type: "image_url", image_url: { url: dataUrl } }]` (ou équivalent selon l’API utilisée).

- **`server/index.js`**  
  - Nouvel endpoint `POST /api/report/legendes` : body `{ image: string (data URL base64), extraction: object }` ; retourne `{ legendes: Array<{ label: string, fleche: { x1, y1, x2, y2 } }> }`.  
  - Optionnel : exposer les légendes dans le flux SSE `understand-stream` (nouvel event `legendes`) lorsque une image est fournie, pour éviter un second appel.

- **`server/pipeline.js`**  
  - Optionnel : si les légendes sont intégrées au stream, ajouter une fonction `runLegends(imageBase64, extraction)` qui appelle le LLM légendes et retourne le tableau ; l’orchestrateur du stream l’appelle après vulgarisation (ou en parallèle) et envoie l’event `legendes`.

### Frontend – types et état

- **`src/types/report.ts`**  
  - Définir `LegendAnnotation` : `{ label: string; fleche: { x1: number; y1: number; x2: number; y2: number } }` (coordonnées normalisées 0–1).  
  - Étendre `ReportPipelineResult` (ou le type partiel utilisé par le panneau) pour inclure `imageForLegends?: string` (URL blob ou data URL) et `legendes?: LegendAnnotation[]`.

- **`src/pages/Index.tsx`**  
  - Conserver une référence à l’image lorsque l’utilisateur charge une photo/radio/IRM (blob URL ou base64) en plus du texte envoyé au pipeline. Passer cette référence (et plus tard les légendes) au panneau droit (ChatPanel / composant d’explication).  
  - Lorsque le rapport provient d’une image : après réception de l’extraction (ou à la fin du stream), appeler `POST /api/report/legendes` avec l’image et l’extraction, puis stocker le résultat dans l’état (ex. `legendes` dans le résultat du pipeline ou état dédié).

- **`src/components/PdfViewer.tsx`**  
  - Exposer vers le parent non seulement `onUnderstandReport(reportText)` mais aussi la source « image » quand le document est une image (ex. callback `onImageSource?.(blobUrlOrDataUrl)` ou remonter l’URL via un state partagé). Le parent (Index) doit pouvoir récupérer l’image pour l’afficher et pour l’appel légendes.

### Frontend – affichage

- **`src/components/ReportExplanationPanel.tsx`** (ou nouveau composant dédié)  
  - Afficher la section « Graphique légendé » lorsque `imageForLegends` et (optionnel) `legendes` sont présents.  
  - Afficher l’image dans un conteneur avec position relative ; en overlay SVG (position absolute, même dimensions que l’image affichée), pour chaque entrée de `legendes` :  
    - Ligne/flèche de `(x1 * width, y1 * height)` à `(x2 * width, y2 * height)` (en pixels affichés).  
    - Label (texte) positionné à l’extrémité de la flèche (x2, y2) ou à côté, avec style lisible (fond, police).  
  - Gérer le redimensionnement : les coordonnées étant normalisées (0–1), le calcul pixel se fait à partir des dimensions réelles du conteneur image (clientWidth/clientHeight ou getBoundingClientRect).

- **Composant réutilisable « ImageWithLegends »** (recommandé)  
  - Props : `imageUrl: string`, `legendes: LegendAnnotation[]`.  
  - Rendu : `<div className="relative">` avec `<img>` et un `<svg>` en overlay dont la viewBox ou les coordonnées sont dérivées des dimensions du conteneur ; dessiner les segments de flèche (ligne + pointe éventuelle) et les `<text>` pour les labels.  
  - Permet de garder `ReportExplanationPanel` lisible et de réutiliser le bloc ailleurs si besoin.

### Cas PDF sans image

- Si le rapport provient uniquement d’un PDF (pas d’upload d’image séparé), ne pas afficher de bloc « graphique légendé » (ou afficher un message du type « Aucune image à légender »).  
- Extension possible (hors scope initial) : extraire une ou plusieurs pages du PDF en images (ex. via `pdfjs-dist` render sur canvas) et les envoyer au même flux légendes ; à traiter en phase ultérieure si nécessaire.

## Algorithmes

### Génération des légendes par le LLM

1. **Entrée**  
   - Image (base64, data URL) du graphique (radio/IRM).  
   - Extraction structurée du rapport (localisation, type_examen, faits_principaux, conclusion_rapport, termes_techniques).

2. **Prompt**  
   - Système : décrire le rôle (assistant qui pose des légendes pédagogiques sur une image d’imagerie médicale pour aider le patient à comprendre ; pas de diagnostic).  
   - Utilisateur : fournir l’image + l’extraction ; demander de renvoyer **un seul objet JSON** contenant une liste `legendes`. Chaque élément : `label` (texte court vulgarisé) et `fleche` avec `x1, y1, x2, y2` (coordonnées normalisées entre 0 et 1 : origine = coin supérieur gauche de l’image, x vers la droite, y vers le bas). La flèche pointe vers la zone à désigner ; (x1,y1) = départ, (x2,y2) = pointe.  
   - Préciser : « Réponds UNIQUEMENT avec un objet JSON valide, sans texte avant ou après. »

3. **Sortie**  
   - Parser la réponse (enlever éventuels blocs markdown ```json) et valider le schéma. Si invalide, retourner un tableau vide ou une valeur par défaut pour ne pas casser l’UI.

4. **Modèle**  
   - Utiliser un modèle avec **vision** (ex. GPT-4o-mini, Gemini avec image) pour que le LLM puisse proposer des coordonnées cohérentes avec le contenu de l’image.

### Affichage des flèches et labels

1. Conteneur : `position: relative`, dimensions définies par l’image (object-contain pour garder le ratio).  
2. Mesurer les dimensions affichées de l’image (après layout) pour faire coïncider l’overlay SVG (même largeur/hauteur).  
3. Pour chaque `legendes[i]` :  
   - `px1 = x1 * width`, `py1 = y1 * height`, `px2 = x2 * width`, `py2 = y2 * height`.  
   - Dessiner une ligne de (px1, py1) à (px2, py2) ; optionnel : triangle ou marqueur en (px2, py2) pour la pointe.  
   - Afficher `label` près de (px2, py2) (ex. à droite ou en dessous avec un petit décalage pour éviter le chevauchement).

## Détails issus du brief utilisateur

- **Vulgarisation** : les graphiques (radio/IRM) servent à **appuyer** l’analyse fournie par le LLM.  
- **Légendes** : faites par un **LLM** qui **propose les coordonnées de départ et d’arrivée des flèches** (pas seulement du texte ; les coordonnées permettent de dessiner les flèches sur l’image).

## Phases recommandées

- **Phase 1 – Données et API**  
  - Types `LegendAnnotation` et extension du résultat du pipeline (image + legendes).  
  - Prompt + parsing JSON dans `server/prompts.js`.  
  - Endpoint `POST /api/report/legendes` dans `server/index.js` et fonction d’appel LLM (vision) dans `server/llm.js` si besoin d’adapter le format des messages (image + texte).

- **Phase 2A – Remontée de l’image et appel légendes**  
  - PdfViewer : remonter la source image au parent (Index) quand le document est une image.  
  - Index : garder `imageUrl` (ou data URL) en état ; après extraction ou à la fin du stream, appeler `/api/report/legendes` et stocker `legendes` (et l’image) dans l’état passé à ChatPanel / ReportExplanationPanel.

- **Phase 2B – Composant d’affichage**  
  - Composant `ImageWithLegends` : image + overlay SVG avec flèches et labels à partir des coordonnées normalisées.  
  - Intégration dans `ReportExplanationPanel` (ou ChatPanel) : afficher le bloc « Graphique légendé » lorsque l’image et les légendes sont disponibles, à côté ou au-dessus des blocs d’explication existants.

Les phases 2A et 2B peuvent être menées en parallèle après la Phase 1.
